# Suggestions

本文档是未裁决治理提案队列，也承接防复发候选。它不是普通建议箱、灵感池、任务 backlog、错误日志或执行记录。

只记录 AI 自发发现、尚未被用户裁决、且可能改变治理文件长期行为的提案。用户已经明确要求或当场裁决的修改，不进入本文档；直接执行，并在 `LOG.md` 记录。

## 入队条件

防复发候选进入本文档前，应同时满足：

- 可能复发。
- 会影响后续 agent 判断、行动或治理演进。
- 暴露了入口协议、信息路由、长期原则、状态接续或治理触发机制的缺口。
- 需要用户裁决后才能进入正式治理文件。

不满足这些条件的普通错误、当场修好的问题、临时执行过程，不进入本文档。

## 提案格式

```text
时间：
建议目标文件：AGENTS.md / README.md / PRINCIPLES.md / STATE.md / LOG.md / SUGGESTIONS.md
建议类型：自由命名，说明它属于哪类治理改进即可
触发事件：
建议内容：
原因和证据：
影响范围：
用户裁决问题：
状态：待确认 / 已确认 / 已拒绝 / 已执行 / 已替代
替代说明：如状态为已替代，说明被哪次新裁决、记录或提案替代。
```

## 使用规则

- 只有未裁决的治理变更进入本文档。
- 提案可以指向 `AGENTS.md` 或 `.foundation/` 下任一治理文件。
- 提案必须说明目标文件、原因和证据、影响范围，以及需要用户裁决的问题。
- 用户确认后，才能修改正式治理文件。
- 已执行的治理变更应记录到 `LOG.md`。
- 不把本文档当作长期规则来源；正式生效内容应进入对应治理文件。

## 队列处理

- 重要工作收尾、用户询问治理问题、或版本同步前，快速查看是否有待确认提案影响本轮工作。
- 用户确认提案后，把变更增量写入目标治理文件，并将提案状态改为“已执行”。
- 用户拒绝提案后，将状态改为“已拒绝”，保留原因摘要即可。
- 旧建议或旧理解被新裁决替代时，将状态改为“已替代”，保留替代说明，不直接删除。
- 已执行、已拒绝或已替代的提案移入“已处理提案”；“待裁决提案”只保留待确认或已确认但未执行的内容。
- 多条提案冲突或重叠时，先合并判断或请求用户裁决，不要分别硬改。
